============================================================
0) КАК ЧИТАТЬ ЭТОТ ДОКУМЕНТ
============================================================

Этот документ построен как «инженерная книга» по страницам.
Для каждой страницы описано:
1. URL / маршрут и ограничения доступа
2. Какой Blade-файл рендерится
3. Какие данные подаются на страницу
4. Какие UI-блоки есть и зачем они нужны
5. Какие формы/действия отправляются на backend
6. Какие JS-скрипты запускаются
7. Какие CSS-файлы влияют на внешний вид
8. Где технические риски и что улучшить

Документ рассчитан на защиту, ревью и дальнейший рефактор.


============================================================
1) ОБЩАЯ АРХИТЕКТУРА РОУТИНГА
============================================================

Файл: routes/web.php

Ключевой принцип:
- Публичные страницы рендерятся напрямую через view/контроллер.
- После авторизации пользователь редиректится по роли.
- Основной кабинет (студент/преподаватель) собран вокруг единого маршрута:
  /teacher/indes?page=...
  где page определяет, какой внутренний шаблон подключается.

Почему так сделано:
- Быстро и удобно для MVP: меньше отдельных контроллеров/роутов.
- Минус: URL менее «чистые» и масштабирование сложнее, чем при
  явном разделении /student/... /teacher/....


============================================================
2) ГЛОБАЛЬНЫЙ LAYOUT
============================================================

Страница/каркас: resources/views/layouts/app.blade.php

Роль layout:
- общий HTML-скелет сайта
- head + favicon + общие CSS/JS
- верхняя навигация
- слот для контента через @yield('content')

2.1 Head и ресурсы
- meta charset/viewport/csrf-token
- title
- favicon:
  - /public/favicon.png
  - /public/apple-touch-icon.png
- подключение общих стилей:
  - public/css/style.css
  - public/css/layout-app.css
  - public/css/mobile-polish.css
- подключение библиотек AOS/Swiper
- ранний preload темы: public/js/theme-preload.js

2.2 Навигация
Логика зависит от роли:
- гостю: Главная, Каталог, Войти
- студенту: + Курсы (student_dashboard)
- преподавателю: + Курсы преподавателя, Редактор
- админу: + Админ
- авторизованному: ЛК + logout

2.3 Технический смысл
- Layout является единой точкой, где проще всего менять глобальные активы,
  тему, фавикон, навигацию и политику общих скриптов.

2.4 Риски
- Если в layout ломается кодировка/символы, это визуально бьет сразу по всем страницам.
- При слишком большом количестве общих скриптов можно получить лишнюю нагрузку.

4) АВТОРИЗАЦИЯ: LOGIN
============================================================

URL: /login
Blade: resources/views/auth/login.blade.php
CSS: public/css/auth-login.css
POST: route('login')

4.1 Что рендерится
- заголовок, подзаголовок
- форма email/password
- вывод ошибок Laravel validation
- ссылка на регистрацию

4.2 Backend-процесс
- POST /login -> AuthController@login
- throttling: 5 попыток в минуту (middleware throttle:5,1)

4.3 Важные поля
- email (required)
- password (required)
- @csrf обязателен

4.4 Риски
- В файле видны признаки «битой кодировки» в текстовых строках.
  Логика не ломается, но UX и презентация страдают.

============================================================

6.2 Интерактивный квест
URL: /quest
Blade: resources/views/open-day/quest.blade.php
JS: public/js/novel-engine.js + public/js/quest-init.js
API: /api/quest/{code?} (QuestApiController@show)

6.3 DOM-архитектура новеллы
Ключевые узлы:
- #quest-root (хранит data-url на API)
- #quest-image
- #quest-speaker
- #quest-text
- #quest-options
- #quest-dictionary-list
- #quest-progress

6.4 Принцип работы движка
- Загружает сценарий (узлы/сцены)
- Рендерит текущую сцену
- По клику на вариант переносит в следующий узел
- Поддерживает history/back
- Рисует словарь сцены
- Подсвечивает слова в тексте и даёт tooltip перевода
- Обновляет прогресс
- Работает с аудио

6.5 Технические особенности

- В некоторых строках UI снова заметны следы неверной кодировки.
- При невалидном JSON сценария страница может визуально «замереть».

  - teacher-panel-analytics + teacher-panel

- Быстрое развитие MVP в одном контуре.

7.4 Минусы
- Одна точка становится «толстой»: сложнее дебажить и рефакторить.

12) СТРАНИЦА ПРЕПОДАВАТЕЛЯ: РЕДАКТОР КУРСОВ
============================================================

Blade: resources/views/teacher/pages/teacher-panel.blade.php
JS-модули:
- public/js/teacher-indes.bootstrap.js
- public/js/teacher-indes.core.js
- public/js/teacher-indes.actions.js

12.1 Назначение
Полноценный CMS-подобный редактор:
- создание курса
- создание модуля
- создание урока
- редактирование структуры шагов (теория/практика)
- медиа-загрузка
- сохранение/удаление урока

12.2 Структура «по шагам»
Шаг 1: курс
Шаг 2: модуль
Шаг 3: урок

12.3 Типы шагов урока
- theory
- multiple_choice
- fill_blanks
- matching
- word_order
- audio_pick
- flashcards

12.4 API, которыми пользуется редактор
- api.course.add
- api.module.add
- api.lesson.get
- api.lesson.save
- api.lesson.delete
- api.dictionary.add / delete
- teacher.assets.upload

12.5 Клиентская архитектура редактора
bootstrap.js:
- парсит JSON-пейлоад из script#teacher-indes-data
- инициализирует глобальные данные

core.js:
- хранит состояние (currentLessonId/currentSteps/...)
- модалки (input/confirm)
- WYSIWYG helper
- подготовка/нормализация медиа
- рендер preview и steps list

actions.js:
- actionHandlers
- универсальный apiRequest
- bind data-action/data-change-action/data-input-action
- CRUD операции курса/модуля/урока/словаря
- загрузка и синхронизация selector-ов

12.6 Техническая оценка
- Для MVP редактор функционально сильный.
- Логика уже достаточно сложная: это кандидат на компонентный рефактор.

12.7 Риски
- Большой объем imperative JS в глобальной области.
- Слабая типизация состояния.
- Высокая цена регрессии без автотестов UI.

13) СТРАНИЦА ПРОХОЖДЕНИЯ УРОКА
============================================================

Blade: resources/views/teacher/pages/lesson-view.blade.php
CSS: public/css/lesson-modern.css
JS: public/js/lesson-engine.js + inline hooks из page-scripts

13.1 Назначение
- Исполнение учебного сценария урока (теория+задачи)
- параллельный форум/комментарии справа

13.2 Основной UI
- header урока + прогресс + HP
- canvas: theory-block + practice-block
- controls: next/skip
- форум: комментарии, лайки, жалобы, редактирование, пин, lock

13.3 Потоки данных
- lesson данные приходят с сервера
- LessonEngine управляет шагами и прогрессом
- complete_lesson отправляется как action в teacher.indes.handle

13.4 Форумный контур
Формы:
- POST lesson.comment.store
- POST lesson.forum.post.like (AJAX)
- POST lesson.forum.post.report
- POST lesson.forum.post.edit
- POST lesson.forum.post.delete
- POST lesson.forum.post.pin
- POST lesson.forum.lesson.toggleLock

13.5 AJAX-поведение
В partial scripts:
- лайк и отправка комментария обрабатываются через fetch
- при успехе обновляются только нужные части
- fallback: частичный reload форума или full reload

13.6 Риски
- Много серверных форм в одном блоке -> чувствительно к CSRF/валидации.
- При кодировочных ошибках страдает читаемость интерфейса форума.

============================================================
18) КЛЮЧЕВЫЕ ТЕХНИЧЕСКИЕ ПРОБЛЕМЫ НА СЕЙЧАС
============================================================

1. Кодировка текста в ряде Blade/JS строк
- Симптомы: «Р…Р…» и т.п.
- Причина: несогласованное сохранение в UTF-8.
- Что делать: массовая проверка и пересохранение файлов в UTF-8,
  затем ручная проверка всех пользовательских строк.

2. Монолитность teacher/indes
- Удобно для MVP, но увеличивает связанность.
- Рекомендация: постепенно выделять отдельные контроллеры/роуты/вьюхи.

3. Большой объем клиентского JS без типизации
- Риск регрессий при доработках.
- Рекомендация: поэтапный переход на модульный TS/Vite и unit-тесты helpers.

4. Часть вычислений в шаблонах
- Перенести тяжелые вычисления в сервисы/view-model для тестируемости.
